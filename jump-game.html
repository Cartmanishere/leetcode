55. Jump Game
First-principles: from "how far can I go?" to the greedy one-pass solution

We're going to derive the solution from scratch — not just memorize the loop. By the end, you'll understand why tracking a single variable works, why greedy beats DP here, and how to spot the same pattern in other reachability problems.

1. What are we really being asked?

You start at index 0. At each position i, the number nums[i] tells you the maximum distance you can leap forward — but you can choose any smaller jump too. You want to know: can I land on the last index?

Let's think about what makes this possible or impossible:

Two equivalent ways to see the problem

Forward view: From where I am, how far can I reach? At each step, expand my "reachable frontier." If it ever falls short of a position I need to reach → stuck.

Backward view: Starting from the last index, work backwards. Which positions can jump to a position that's already "good"? If index 0 is good → success.

The forward view is simpler — and that's the greedy approach.

Concrete examples:

Example 1: [2, 3, 1, 1, 4] — starting with 2 at index 0, you can reach indices 1 and 2. From index 1 (value 3), you can reach all the way to index 4 (the end). ✅ Reachable.

Example 2: [3, 2, 1, 0, 4] — from index 0 (3) you can reach up to index 3. But at index 3, the value is 0 — you can't move further. No matter which path you take, you get trapped at index 3. ❌ Unreachable.

The "aha" moment

You don't need to plan a full path. You just need to know: as you scan left to right, is there any point where the furthest reachable index falls behind your current position?

If maxReach < i at any point → you're stuck. Otherwise, keep going and update maxReach = max(maxReach, i + nums[i]).

2. The greedy recurrence

Define maxReach = the furthest index you can reach so far.

Initial state: maxReach = 0 // Initially, you can only reach index 0 (your start) At each position i: if maxReach < i → return false // You can't even get to position i maxReach = max( maxReach, i + nums[i] ) └─ stay where I am └─ jump from i as far as I can If we finish the loop without getting stuck: return true

Break this down:

  • Stuck check: if the furthest you can reach (maxReach) is behind the position you're at (i), you physically cannot be at position i. Game over.
  • Reach expansion: from position i, you can jump up to i + nums[i]. Update maxReach if this extends your frontier.
  • Implicit win: if you make it through the entire array without getting stuck, you can reach the last index. (Actually, you can early-exit as soon as maxReach ≥ last_index.)

Why this is greedy (and why greedy works here)

At each step, you take the locally optimal action: jump as far as you can from wherever you are. But unlike many greedy problems, there's no tradeoff — reaching further can never hurt you. Here's why:

  1. If you can reach index j, you can also reach every index between your current position and j (just take smaller jumps).
  2. So "can reach" is monotonic — from index i, you create a continuous reachable interval [i, i+nums[i]].
  3. The union of reachable intervals from all visited positions is [0, maxReach] — a single contiguous block from the start.
  4. Since maxReach only grows, there's never a reason to be "strategic" about which position to jump to — farther is always better.

This monotonicity is what makes greedy optimal. If jumps were exact (not "up to"), or if there were costs, greedy would fail — and that's when DP takes over.

The key insight: the question "can I reach position i?" is answered by one value: maxReach. We don't need to try every path (O(2ⁿ)) or build a DP array of reachable positions (O(n²·m)). One scan, one variable — O(n) time, O(1) space.

4. Why the greedy one-pass is optimal

The key property that makes greedy correct here is monotonicity of reachability.

Proof sketch

Claim: After processing positions [0..i], the set of reachable positions is exactly [0..maxReach] — a single contiguous prefix.

Base case (i=0): You can reach [0, nums[0]] — a contiguous segment from 0.

Inductive step: Assume after processing [0..i-1], reachable = [0..R] where R = maxReach. At position i:

  • If i > R: position i is unreachable → we're stuck.
  • If i ≤ R: we can be at i, and from i we can reach up to i + nums[i]. So the new reachable set is [0..max(R, i + nums[i])] — still a contiguous prefix.

This invariant proves: maxReach is sufficient to track everything. No DP needed.

When greedy fails: if the problem asked for the minimum number of jumps (LeetCode 45 — Jump Game II), greedy on maxReach alone doesn't work. You need to track "when does the current jump's range end?" — which adds a second state variable. Even then, it's still greedy, but the decision is more subtle.

5. The complete solutions

5a. Greedy — O(n) time, O(1) space (optimal)

/** * Jump Game — Greedy * Track the furthest reachable index as we scan. * If we ever encounter an index beyond maxReach → stuck. */ function canJump(nums) { let maxReach = 0; for (let i = 0; i < nums.length; i++) { // Can I even be at this position? if (maxReach < i) return false; // Extend my reach maxReach = Math.max(maxReach, i + nums[i]); // Early exit: already can reach the end if (maxReach >= nums.length - 1) return true; } return true; }

5b. Optimized Greedy — loop only while reachable

/** * Jump Game — Alternative greedy phrasing * Loop condition itself encodes the "stuck" check: * - Continue only while i ≤ maxReach (i is reachable) * - If loop finishes naturally and maxReach ≥ last index → success */ function canJump(nums) { let maxReach = 0; for (let i = 0; i <= maxReach; i++) { maxReach = Math.max(maxReach, i + nums[i]); if (maxReach >= nums.length - 1) return true; } return false; // maxReach < n-1 and we can't advance further }

5c. DP — O(n²) time, O(n) space (for learning)

/** * Jump Game — DP * dp[i] = true if index i is reachable from the start * * Recurrence: * dp[i] = true if there exists j < i such that * dp[j] == true AND j + nums[j] >= i * * This is the "backward look" approach — at each i, look * at all previous positions to see if any can reach i. */ function canJumpDP(nums) { const n = nums.length; const dp = new Array(n).fill(false); dp[0] = true; // Base case: we start at index 0 for (let i = 0; i < n; i++) { if (!dp[i]) continue; // Can't propagate from an unreachable position // Mark all positions reachable from i const maxJump = Math.min(n - 1, i + nums[i]); for (let j = i + 1; j <= maxJump; j++) { dp[j] = true; if (j === n - 1) return true; // Early exit } } return dp[n - 1]; }

Greedy Approach

  • One variable: maxReach
  • O(n) time, O(1) space
  • Exploits the contiguous reachable interval property
  • Standard interview answer

DP Approach

  • Explicit dp[i] = "is i reachable?"
  • O(n²) time, O(n) space — too slow for n=10⁴
  • Easier to derive if you think "backwards from the answer"
  • Useful stepping stone — then optimize to greedy

6. The backward DP (an alternative lens)

There's a third approach that's also O(n) — scanning right to left:

Backward DP / "good position" approach

Define lastGood = the rightmost index we know can reach the end. Initially, lastGood = n - 1 (the last index can trivially reach itself).

Scan from right to left. At position i, ask: can I jump from i to anywhere ≥ lastGood? If i + nums[i] ≥ lastGood, then i is also "good" — set lastGood = i.

At the end: if lastGood === 0 → we can reach the end from the start.

/** * Jump Game — Backward DP * O(n) time, O(1) space */ function canJumpBwd(nums) { let lastGood = nums.length - 1; for (let i = nums.length - 2; i >= 0; i--) { if (i + nums[i] >= lastGood) { lastGood = i; // This position can reach a "good" position } } return lastGood === 0; }
The backward scan is elegant and also O(n) / O(1). It's the "reverse" of the forward greedy. Choose whichever feels more natural — in an interview, showing either one demonstrates solid reasoning. The backward version is especially nice when the follow-up is "find the minimum number of jumps" (Jump Game II), because the "good position" concept generalizes well.

The common thread: a monotonically expanding frontier where "further is always better." When you spot that property, greedy is on the table — and it's usually the optimal answer.

8. First-principles derivation checklist

When you see a "can you reach the end?" problem, ask:

  1. Is reachability monotonic? — Can I reach every position between my start and the furthest reachable? If yes, the reachable set is a contiguous prefix → one variable suffices.
  2. What's the state? — For Jump Game I, it's just maxReach. For Jump Game II (minimum jumps), you need maxReach + currentEnd + jumps. The complexity of the state tells you whether it's simple greedy or layered greedy.
  3. Does "further" always help? — If there's a cost, penalty, or diminishing return associated with jumping far, greedy fails. That's your cue to reach for DP.
  4. Can I early-exit? — As soon as maxReach ≥ lastIndex, return true. As soon as i > maxReach, return false. These early exits often speed up real-world performance dramatically on sparse inputs.

🔴 Blocking Bugs

Bug #1 — max_reach >= len(nums) is an off-by-one error BLOCKING

The last index of an array of length n is n - 1, not n. By comparing against len(nums) instead of len(nums) - 1, your early exit never fires when the answer is True — because max_reach can never exceed the array bounds. When max_reach == n - 1 (exactly at the last index), the condition max_reach >= n is False, so it keeps scanning — and eventually hits the return False fallback.

Counterexample: [2, 3, 1, 1, 4] — the main LeetCode example. Expected: True. Your code: False.

At idx=1, max_reach becomes 4, which equals len(nums) - 1 = 4. The last index is reachable! But because your check tests max_reach >= 5, it misses this. The code runs to completion and returns False.

The last valid index is n-1. When max_reach covers it, you're done.

Bug #2 — return False at the end is logically backwards BLOCKING

With Bug #1 fixed, the single-element case would work. But conceptually, the return False is wrong. Think about what it means for the loop to complete without returning:

  • The stuck check (max_reach < idx) never fired → we were never stuck at any position.
  • The early exit (max_reach >= n - 1) never fired → max_reach never covered the last index.

But if we processed every index without getting stuck, that means we visited every position including the last one. By definition, the last index is reachable. The correct fallback after a complete scan is return True.

⚠️ In practice, with Bug #1 fixed and the early exit working, this return False would only be hit on edge cases where the early exit somehow never fires despite reaching the end. But semantically return True is the correct invariant, and writing return False signals to any reviewer that you haven't thought through what "loop completed" actually means.
