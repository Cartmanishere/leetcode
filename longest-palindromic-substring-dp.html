Longest Palindromic Substring
Mastering the DP recurrence pattern

📋 Problem Statement

Given a string s, return the longest palindromic substring in s.

Example 1

Input: s = "babad" Output: "bab" Explanation: "aba" is also a valid answer.

Example 2

Input: s = "cbbd" Output: "bb"

Constraints

  • 1 ≤ s.length ≤ 1000
  • s consists of only digits and English letters.

We're going to build the intuition from the ground up — not just memorize the solution. By the end, you'll know how to derive the recurrence for any "optimal substring" problem.

1. What makes a palindrome?

A string s is a palindrome if it reads the same forwards and backwards:

r
a
c
e
c
a
r

"racecar" — outer chars match, and the inside is also a palindrome.

This is the recurrence in disguise:

The core insight

s[i..j] is a palindrome iff:

  1. s[i] === s[j] — the outer characters match, AND
  2. s[i+1 .. j-1] is a palindrome — the inner substring is also a palindrome.

Base cases: every single character is a palindrome. Two identical adjacent characters are a palindrome.

2. The recurrence relation

Let dp[i][j] = true if s[i..j] is a palindrome dp[i][j] = (s[i] == s[j]) && (j - i < 2 || dp[i+1][j-1]) └─ base: length 1 or 2 ─┘ └─ recurse to inside ─┘

Break this down:

  • j - i < 2 covers lengths 1 (i==j) and 2 (j==i+1). No inner substring needed — just check s[i]==s[j].
  • For length ≥ 3, we depend on the answer for a shorter substring — the classic DP hallmark.

Why this pattern is reusable

Any problem where the answer for [i..j] depends on [i+1..j-1] follows this exact shape:

dp[i][j] = f( s[i], s[j], dp[i+1][j-1] )

This shows up in: Palindrome Partitioning, Longest Palindromic Subsequence, Minimum Insertions to Form Palindrome, and many "interval DP" problems.

The key to recognizing DP applicability: the answer for a larger interval is built from the answer for a smaller interval. You fill the DP table diagonally — by increasing substring length.

4. How to fill the DP table (the traversal order)

You cannot fill row-by-row left-to-right, because dp[i][j] depends on dp[i+1][j-1] (the cell below-left).

Strategy: Fill by substring length.

The diagonal traversal

T
T
T
T
T

Pass 1: length=1 (diagonal)

Pass 2: length=2 (one off diagonal)

Pass 3: length=3 (two off diagonal)

... and so on until length=n.

// Fill by length for (let len = 1; len <= n; len++) { for (let i = 0; i + len - 1 < n; i++) { let j = i + len - 1; // When filling [i][j], dp[i+1][j-1] is already computed // because i+1..j-1 has length len-2 < len } }
This "diagonal fill" pattern is universal for interval DP. Whenever dp[i][j] depends on a strictly shorter interval, iterating by length guarantees correct dependency order.

5. The complete solution

/** * Longest Palindromic Substring — DP O(n²) time, O(n²) space * Pattern: Interval DP with dp[i][j] = (s[i]==s[j]) && (len<3 || dp[i+1][j-1]) */ function longestPalindrome(s) { const n = s.length; const dp = Array.from({ length: n }, () => new Array(n).fill(false)); let start = 0, maxLen = 1; // Base: every single char is a palindrome for (let i = 0; i < n; i++) dp[i][i] = true; // Fill by increasing substring length for (let len = 2; len <= n; len++) { for (let i = 0; i + len - 1 < n; i++) { const j = i + len - 1; if (s[i] !== s[j]) { dp[i][j] = false; // Mismatch → not a palindrome } else if (len <= 3) { dp[i][j] = true; // Length 2 or 3 with matching ends } else { dp[i][j] = dp[i+1][j-1]; // Recurrence: check inner substring } if (dp[i][j] && len > maxLen) { start = i; maxLen = len; } } } return s.substring(start, start + maxLen); }
Space optimization: Since dp[i][j] only depends on row i+1, you can reduce to O(n) space with two rows. But for learning, the full table is clearer.

6. Recognizing this pattern in other problems

Problem Recurrence Traversal
Longest Palindromic Substring dp[i][j] = s[i]==s[j] && dp[i+1][j-1] By length
Longest Palindromic Subsequence dp[i][j] = s[i]==s[j] ? 2+dp[i+1][j-1] : max(dp[i+1][j], dp[i][j-1]) By length
Palindrome Partitioning II (min cuts) Use palindrome DP table + min-cut DP By length
Stone Game / Burst Balloons dp[i][j] = max(dp[i][k] + dp[k][j] + ...) By length

The common thread: interval DP — the answer for [i..j] splits into smaller intervals. Always fill by increasing length.

🟡 Design / Code Quality Issues

Issue #4 — Edge case returns wrong sentinel

if len(s) < 1: return '0' — the character '0' is a valid English letter/digit and could appear in the input. An empty string should return "". Returning '0' is misleading and would fail test cases where the input is empty (though constraints say len≥1, it's sloppy).

Issue #5 — No base case for length 1

You never initialize dp[i][i] = True. The code works around this by special-casing lengths 2 and 3, but this is fragile. If you later refactored to unify the loops, the recurrence dp[i+1][j-1] would fail for substrings where the inner portion is length 1 (because hash_array_get returns False for unset keys).

Missing base cases = missing the foundation of DP. The interviewer wants to see you explicitly set up the base.

Issue #6 — Unnecessary special-casing of lengths 2 and 3

You have three separate loops (len=2, len=3, len≥4) when the general recurrence handles all of them. Length 2 (j-i+1=2): the inner substring dp[i+1][j-1] doesn't exist (i+1 > j-1), so the recurrence needs a guard. Length 3: the inner substring is length 1, which is always a palindrome, so s[i]==s[j] suffices. But all this collapses into one loop with the len <= 3 guard — exactly what the canonical solution does.

The unified approach:
if s[i] == s[j] and (len <= 3 or dp[i+1][j-1])
Issue #7 — Anti-pattern: try/except assert for dict defaults

hash_array_set uses assert dp[i] inside a try/except to detect missing keys. This is non-idiomatic and confusing:

  • assert can be disabled with python -O, silently breaking the code
  • The idiomatic way is dp.setdefault(i, {}) or if i not in dp: dp[i] = {}
  • hash_array_get has an unused parameter val — dead code
Issue #8 — Type hints are wrong

dict[int, int] suggests values are int, but you're storing nested dicts of bools. The correct hint is dict[int, dict[int, bool]]. Wrong type hints signal to the interviewer that you're not thinking about the data structure's shape.

Issue #9 — Sparse hash-map buys nothing here

The idea of using a hash-map to "avoid storing all the data" is premature optimization. For n≤1000, an n×n list-of-lists uses ~8MB for booleans — negligible. The hash-map approach adds overhead (hashing, collisions), makes traversal harder, and doesn't reduce asymptotic complexity. The real space optimization is the two-row rolling array that drops O(n²)→O(n).

You understood the recurrence perfectly — this is the hard part. The intuition that dp[i][j] = s[i]==s[j] and dp[i+1][j-1] and that you fill by increasing length is exactly right. The hash-map idea shows you're thinking about space. The separate handling of base lengths shows correct DP thinking.

🎯 The deeper pattern across all three submissions

All three submissions share the same root cause: you're treating str_len as substring length when it's actually j - i (length − 1). This creates a cascade: the range bound, the base-case guard, and the recurrence all operate on the wrong quantity.

Tweaking ranges and start values won't fix it. You need to rename the variable to match reality and derive j from the invariant:

# INVARIANT: j - i + 1 == length # Therefore: j = i + length - 1 for length in range(2, n + 1): for i in range(n - length + 1): j = i + length - 1 dp[i][j] = (s[i] == s[j]) and (length <= 3 or dp[i+1][j-1])

Once j is correct, everything collapses into place: the range bound, the guard, and the recurrence all use the same length variable that means exactly what it says.

When you see a substring/subarray problem, ask:

  1. Can the answer for [i..j] be expressed in terms of [i+1..j], [i..j-1], or [i+1..j-1]?
  2. Are the base cases (length 0, 1, or 2) trivial?
  3. Is the recurrence monotonic (always depends on shorter intervals)?

If yes to all three → interval DP, fill by length.
