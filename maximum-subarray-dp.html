Maximum Subarray
First-principles DP: from "what should I do at each step?" to Kadane

We're going to derive the recurrence from scratch — not just memorize Kadane's algorithm. By the end, you'll know why the recurrence takes this shape and how to recognize the same pattern in other problems.

1. What is a subarray? What are we really optimizing?

A subarray is a contiguous segment of the array. Unlike a subsequence, you cannot skip elements.

We want the subarray with the maximum sum. Let's think about what that means one element at a time:

First-principles question

Imagine you're scanning left to right. When you reach position i, you look at nums[i] and ask:

"Should I extend the best subarray ending before me,
or should I start fresh right here?"

This is the only choice at each position — and it's what makes DP work.

Let's make this concrete with [-2, 1, -3, 4, -1, 2, 1, -5, 4]:

At index 0: only one option — take -2. Best subarray ending at 0 has sum -2.

At index 1 (value 1): extend with -2 → sum -1, or start fresh → sum 1. Start fresh wins.

At index 3 (value 4): extend the best ending at 2 (-2) → sum 2, or start fresh → sum 4. Start fresh wins.

At index 4 (value -1): extend 4 → sum 3, or start fresh → sum -1. Extend wins.

The "aha" moment

When is it always better to start fresh? When the previous best is negative — adding a negative to the current number can only make it smaller. When the previous best is positive, extending always improves the sum.

This gives us a simple rule:
if previous_sum > 0 → extend it. Otherwise → start fresh.

2. The recurrence relation

Define dp[i] = maximum sum of a subarray ending at index i.

Base case: dp[0] = nums[0] // the only subarray ending at index 0 Recurrence (i ≥ 1): dp[i] = max( nums[i], dp[i-1] + nums[i] ) └─ start fresh ─┘ └── extend previous ──┘ Final answer: result = max( dp[0], dp[1], ..., dp[n-1] )

Break this down:

  • start fresh: throw away everything before i. The best subarray ending at i is just nums[i] alone.
  • extend: take the best subarray ending at i-1 and append nums[i]. This is only beneficial when dp[i-1] > 0.
  • The recurrence max(nums[i], dp[i-1] + nums[i]) encodes both cases in one line — extend wins when dp[i-1] > 0, start fresh wins otherwise.

Why this recurrence has optimal substructure

To find the best subarray ending at i, we only need:

  1. nums[i] — the current element (given)
  2. dp[i-1] — the best subarray ending at the previous position (already computed)

We don't need to know which subarray produced dp[i-1] — the sum alone is enough. This is the hallmark of 1D DP: each state depends only on the immediately previous state.

The key DP insight: we don't try every possible subarray (that would be O(n²)). Instead, we reuse the answer for subarrays ending at i-1 to compute the answer for subarrays ending at i in O(1). The answer for the whole array is just the maximum over all dp[i].

4. The traversal order (why it's simple)

Unlike interval DP (palindromic substring), this is a 1D DP. The traversal is straightforward:

Left-to-right, single pass

dp[i] depends only on dp[i-1]. There's no dependency on future indices or shorter intervals. You can fill the array in a single left-to-right pass — O(n) time.

// Traversal: one pass, left to right for (let i = 1; i < n; i++) { dp[i] = Math.max(nums[i], dp[i-1] + nums[i]); // dp[i-1] is already computed → safe }
Because each dp[i] depends only on dp[i-1], we can drop the array entirely and keep a single variable. This is Kadane's algorithm — same recurrence, O(1) space.

5. The complete solutions

5a. Full DP array — O(n) time, O(n) space

/** * Maximum Subarray — DP array * dp[i] = max sum of subarray ending at i * Recurrence: dp[i] = max(nums[i], dp[i-1] + nums[i]) */ function maxSubArray(nums) { const n = nums.length; const dp = new Array(n); dp[0] = nums[0]; // Base case let maxSum = dp[0]; for (let i = 1; i < n; i++) { dp[i] = Math.max(nums[i], dp[i-1] + nums[i]); maxSum = Math.max(maxSum, dp[i]); } return maxSum; }

5b. Kadane's Algorithm — O(n) time, O(1) space

/** * Maximum Subarray — Kadane's Algorithm * Space-optimized: track only current dp value */ function maxSubArray(nums) { let currentMax = nums[0]; // dp[i] — best ending at current position let globalMax = nums[0]; // max(dp[0]..dp[i]) for (let i = 1; i < nums.length; i++) { currentMax = Math.max(nums[i], currentMax + nums[i]); globalMax = Math.max(globalMax, currentMax); } return globalMax; }

DP Array Version

  • Explicit dp[i] semantics
  • Easier to reason about
  • Extensible (e.g., reconstruct the subarray)
  • O(n) space

Kadane's Version

  • Same recurrence, different bookkeeping
  • currentMax = dp[i] at step i
  • O(1) space — optimal
  • Standard interview answer

6. Divide and Conquer (the follow-up)

The problem's follow-up asks for a divide and conquer solution. Here's the intuition:

The cross-midpoint insight

For any subarray, one of three things is true:

  1. It lies entirely in the left half
  2. It lies entirely in the right half
  3. It crosses the midpoint (starts in left, ends in right)

Cases 1 and 2 are recursive. Case 3 can be computed in O(n) by finding the max suffix sum of the left half and max prefix sum of the right half, then adding them.

/** * Maximum Subarray — Divide & Conquer * T(n) = 2T(n/2) + O(n) → O(n log n) */ function maxSubArrayDC(nums, left = 0, right = nums.length - 1) { if (left === right) return nums[left]; const mid = Math.floor((left + right) / 2); // Best in left half, best in right half const leftBest = maxSubArrayDC(nums, left, mid); const rightBest = maxSubArrayDC(nums, mid + 1, right); // Best crossing the midpoint let leftSum = -Infinity, sum = 0; for (let i = mid; i >= left; i--) { sum += nums[i]; leftSum = Math.max(leftSum, sum); } let rightSum = -Infinity; sum = 0; for (let i = mid + 1; i <= right; i++) { sum += nums[i]; rightSum = Math.max(rightSum, sum); } const crossBest = leftSum + rightSum; return Math.max(leftBest, rightBest, crossBest); }
The divide-and-conquer approach runs in O(n log n) — slower than Kadane's O(n). However, the cross-midpoint pattern (left suffix max + right prefix max) is a classic technique that shows up in segment trees and range query problems where you do need it.

7. Recognizing this pattern in other problems

Problem Recurrence Pattern
Maximum Subarray dp[i] = max(nums[i], dp[i-1] + nums[i]) 1D, depends on dp[i-1]
Maximum Product Subarray dpMax[i] = max(nums[i], dpMax[i-1]*nums[i], dpMin[i-1]*nums[i]) 1D, track both min and max
Maximum Sum Circular Subarray Kadane + Kadane on negated array + wrap-around 1D + circular trick
Best Time to Buy/Sell Stock dp[i] = max(dp[i-1], price[i] - minSoFar) 1D, depends on prefix min
House Robber dp[i] = max(dp[i-1], dp[i-2] + nums[i]) 1D, depends on dp[i-1] and dp[i-2]

The common thread: 1D DP with a "continue-or-restart" decision. The recurrence is always dp[i] = max(fresh_start, extend_previous) — the only difference is what "extend" means in each context.

8. First-principles derivation checklist

When you see a "contiguous subarray" problem, ask:

  1. What does "ending at i" look like? — Define dp[i] as the best answer for subarrays that must end at i. This constraint is what unlocks the recurrence.
  2. What's the choice at each position? — For maximum subarray, it's "extend vs. fresh." For product subarray, it's "extend positive, extend negative, or fresh." The choice defines the recurrence.
  3. How many previous states do I need? — If dp[i] depends only on dp[i-1], you have O(1) space. If it depends on dp[i-2] or a window, you need more.
  4. Is the answer dp[n-1] or max(dp[0..n-1])? — For "ending at" DP, the answer is typically the maximum over all dp[i], not necessarily the last one.

🔴 Blocking Bugs

Bug #1 — max_sum = n destroys the global maximum BLOCKING

Every time cur_sum < 0, you reset max_sum = n without comparing against the existing best. This means if the previously found global maximum is larger than the fresh-start element, you throw it away.

Counterexample: [2, -3, 1]

Step idx nums[idx] cur_sum (before) Branch max_sum (after) Correct max
init 0 2 — — 2 2
1 1 -3 2 else (extend) 2 2
2 2 1 -1 if (fresh) 1 2

At idx=2, cur_sum = -1 < 0 → enters the fresh branch → max_sum = 1. The correct answer is 2 (the element at index 0 alone). Your code returns 1.

⚠️ This bug is easy to miss because it only fires when: (a) cur_sum goes negative, AND (b) the fresh-start element is smaller than a previous best. Many test cases never trigger (a) because they're overwhelmed with positive numbers.

Fix:
max_sum = max(max_sum, n) — the global max should never decrease
Bug #2 — a_start / a_end corrupted by the same overwrite BLOCKING

When Bug #1 fires, the tracking variables a_start and a_end are reset to the fresh-start position even though that new subarray doesn't beat the old best. This means:

  • The print at the bottom reports the wrong subarray
  • If anyone extended this code to return the subarray (instead of just the sum), it would be silently wrong
  • The tracking variables disagree with max_sum — a classic invariant violation

In the [2, -3, 1] counterexample: a_start=2, a_end=2 but max_sum should be 2 (from index 0). The tracking says [1] is the answer, which contradicts the correct answer [2].

Fix:
Only update a_start/a_end when the fresh-start actually beats the existing max_sum:
if n > max_sum: a_start = a_end = idx; max_sum = n

🟡 Design / Code Quality Issues

Issue #3 — Debug print statements in production submission

LeetCode doesn't penalize print(), but in a real interview or code review, debug scaffolding left in the final answer signals carelessness. If the problem asked for the subarray itself, the prints would be a substitute for a proper return value — which isn't what was asked.

Issue #4 — Variable n shadows conventional meaning

n is universally used for array length. Using it for the current element is confusing — especially when len(nums) appears nearby. An interviewer will ask: "what does n represent here?" Use num or x.

Issue #5 — Inverted condition structure

You test if cur_sum < 0 (the bad case) first and else (the good case) second. The canonical Kadane tests if cur_sum > 0 (extend) / else (fresh). Your version is functionally equivalent but reads as "when things are bad, do X; otherwise, Y" — which forces the reader to invert the logic mentally. Always put the affirmative branch first:

# Canonical: test the good case first if cur_sum > 0: cur_sum += num # extend — the natural, desirable action else: cur_sum = num # start fresh — the fallback max_sum = max(max_sum, cur_sum)
Issue #6 — Ambiguous comment indentation inside else:

This is subtle but it's exactly the kind of thing a strict reviewer flags:

else: # if it's positive - then include it new_sum = cur_sum + n

The comment is indented to the same level as else: (12 spaces), while new_sum is at 16 spaces. Python technically accepts this — comments are ignored by the parser and don't affect block structure. But every human reader will do a double-take wondering if this is a syntax error. The comment should be inside the block:

else: # cur_sum >= 0 — extending is beneficial or neutral new_sum = cur_sum + n
Issue #7 — a_start never updated during extension

When the else branch finds new_sum > max_sum, you update a_end = idx but never touch a_start. This happens to work because a_start was set when you entered the current subarray (the last time you started fresh). But this is an implicit invariant — the code doesn't state it, so a future refactor could break it. Explicitly tracking where the current subarray begins (separate from where the best subarray begins) is clearer:

curr_start = 0 # where the current running subarray starts for idx, num in enumerate(nums[1:], 1): if cur_sum > 0: cur_sum += num else: cur_sum = num curr_start = idx # fresh subarray begins here if cur_sum > max_sum: max_sum = cur_sum a_start = curr_start a_end = idx
Issue #8 — Subarray tracking is premature for the problem

LeetCode 53 asks for the sum only. Adding a_start, a_end, and print() to reconstruct the subarray is a feature the problem doesn't ask for. In an interview, this is neutral-to-positive if you get it right. But when tracking variables contain bugs (Issue #2, #7), the extra complexity becomes a liability — it introduces failure modes for something that wasn't required. Ship the simplest correct solution first, then extend.
